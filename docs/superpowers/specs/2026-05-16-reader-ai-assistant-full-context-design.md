# 阅读器 AI 助手与整本优先上下文设计

## 背景

当前阅读器只负责加载小说章节、目录、搜索、阅读设置和阅读进度。AI 能力集中在小说详情页和 AI 配置页：角色分析、设定提取、写作风格分析都会读取小说文件片段后调用当前激活的 AI 配置。

现有小说分析并没有真正读取整本小说。后端 `extract_text_excerpt(file_path, max_chars=12000)` 只读取文件开头部分，因此长篇小说的后续角色、设定、风格变化不会进入分析上下文。用户希望阅读器页面增加 AI 助手，并且阅读器助手与现有小说分析都改为“整本小说优先”：短篇尽量完整进入 AI 上下文，长篇则用覆盖全书范围的抽样上下文，而不是只取开头。

## 目标

1. 在阅读器页面新增 AI 助手，用户可以围绕当前阅读的小说提问。
2. 阅读器助手使用当前小说作为上下文，并额外关注当前阅读章节。
3. 将角色分析、设定分析、写作风格分析改为共享同一套“整本优先”上下文构建逻辑。
4. 短篇小说在上下文预算内时完整发送给 AI。
5. 长篇小说超过预算时，使用开头、结尾、中间均匀片段和当前章节优先的抽样上下文，覆盖全书范围。
6. 保持现有 AI 配置、错误处理和分析结果持久化模型不做大范围重构。
7. 用 Python 后端测试和 Node 静态 UI 测试覆盖新增行为。

## 非目标

1. 不做向量数据库、嵌入检索或长期索引。
2. 不做长篇小说多轮分块总结和再汇总。
3. 不把阅读器助手对话记录写入数据库。
4. 不改 AI 配置页的供应商配置方式。
5. 不改变小说文件解析、章节识别和阅读进度的既有契约。

## 推荐方案

采用共享全书上下文构建器。

后端新增一个可复用 helper，例如 `build_novel_ai_context(...)`。它负责读取小说元数据、标签、TXT/EPUB 章节文本，并根据预算返回结构化上下文结果。阅读器助手、角色分析、设定分析、写作风格分析都调用它，避免每个功能各自实现“整本优先”的判断。

不采用前端拼上下文，因为阅读器页面只能可靠获得当前章节，无法统一处理 EPUB、标签、文件读取失败和长篇抽样。不采用分块多轮分析，因为它会显著增加耗时、成本和失败面，适合以后作为“深度分析”单独设计。

## 上下文构建

新增共享上下文结果结构：

```python
{
    "metadata_text": "...",
    "content_text": "...",
    "context_text": "...",
    "is_full_text": True,
    "is_truncated": False,
    "source_chars": 73210,
    "included_chars": 73210,
    "chapter_count": 18,
    "segments": [
        {
            "label": "完整正文",
            "chapter_index": 0,
            "chapter_title": "第一章",
            "text": "..."
        }
    ]
}
```

字段含义：

- `metadata_text`：小说标题、作者、简介、分类、标签。
- `content_text`：实际提供给 AI 的正文内容。
- `context_text`：元数据加正文的最终上下文文本。
- `is_full_text`：是否包含完整正文。
- `is_truncated`：是否因为预算裁剪。
- `source_chars`：小说正文原始字符数。
- `included_chars`：实际送入上下文的正文字符数。
- `chapter_count`：解析出的章节数。
- `segments`：抽样片段列表，用于状态提示和测试验证。

建议第一版使用明确的字符预算：

- 分析类接口默认正文预算：`60000` 字符。
- 阅读器助手默认正文预算：`60000` 字符。
- 当前章节额外优先预算：包含在总预算内，不额外突破上限。

如果小说全文字符数小于等于预算，`content_text` 使用完整章节正文，并标记 `is_full_text = True`。

如果小说全文超过预算，构建器按以下规则生成片段：

1. 优先保留当前阅读章节，如果调用方传入 `focus_chapter_index`。
2. 保留开头片段，帮助 AI 理解题材、主角和初始设定。
3. 保留结尾片段，帮助 AI 覆盖后期人物和设定变化。
4. 从中间章节均匀抽样若干片段，避免只关注小说开头。
5. 每个片段附带章节序号和章节标题。
6. 若片段合并后仍超过预算，从片段尾部裁剪，保留片段标签。

长篇上下文开头需要明确提示：

```text
以下正文为覆盖全书范围的抽样片段，并非完整全文。只能基于这些片段和元数据回答，不要编造未提供的剧情、人物关系或结局。
```

完整上下文则提示：

```text
以下正文为完整小说正文。回答只能基于提供的小说内容和元数据。
```

TXT 和 EPUB 都通过 `get_cached_reader_file(actual_path)` 读取章节，避免 EPUB 被当作普通文本读取。不可读文件返回空正文，但仍可用标题、作者、简介作为最低限度上下文。

## 阅读器助手体验

阅读器顶部工具栏新增 AI 助手按钮，图标使用现有 Font Awesome 风格。点击后在阅读器右侧打开助手面板；桌面端面板占据阅读器右侧固定宽度，正文区域随之收窄；移动端以覆盖面板打开。

面板包含：

- 上下文状态：例如“已使用完整小说上下文”或“小说过长，已使用全书抽样上下文”。
- 对话记录：用户问题和 AI 回答。
- 输入框：支持多行输入。
- 发送按钮：调用后端接口，等待时禁用。
- 清空按钮：清空当前阅读器会话中的对话。
- 错误提示：展示未配置 AI、文件不可读、模型错误等信息。

助手对话只保存在当前浏览器内存中。关闭阅读器或打开另一部小说时清空，不新增数据库表。

阅读器进入沉浸模式时，助手面板关闭。退出沉浸模式后可重新打开助手。

## 阅读器助手 API

新增接口：

`POST /api/ai/novels/<novel_id>/reader-assistant`

请求体：

```json
{
  "question": "这个角色为什么要隐瞒身份？",
  "chapter_index": 12,
  "chapter_title": "第十三章 雨夜",
  "conversation": [
    {"role": "user", "content": "前面发生了什么？"},
    {"role": "assistant", "content": "前文中..."}
  ]
}
```

行为：

1. 校验小说存在。
2. 校验 `question` 非空，过长问题会裁剪到合理上限，例如 `2000` 字符。
3. 校验已配置可用 AI 客户端。
4. 调用共享上下文构建器，传入 `focus_chapter_index`。
5. 构建 system prompt，要求 AI 只基于当前小说上下文回答。
6. 将最近几轮对话追加到消息中，控制总长度，避免前端传入无限历史。
7. 调用 `client.chat(messages, stream=False)`。
8. 返回 AI 回答和上下文状态。

响应：

```json
{
  "success": true,
  "data": {
    "answer": "从已提供的片段看...",
    "context": {
      "is_full_text": false,
      "is_truncated": true,
      "included_chars": 60000,
      "source_chars": 238120,
      "chapter_count": 76,
      "segment_count": 10
    }
  }
}
```

错误处理：

- 小说不存在：`404`。
- 问题为空：`400`。
- AI 未配置：`400`。
- AI 调用失败：`500`。

## 现有分析改造

以下接口改为使用共享上下文构建器：

- `POST /api/ai/novels/<novel_id>/characters/analyze`
- `POST /api/ai/novels/<novel_id>/settings/analyze`
- `POST /api/ai/novels/<novel_id>/writing-style/analyze`

现有 `build_character_analysis_messages`、`build_setting_analysis_messages`、`build_writing_style_analysis_messages` 接收的正文参数从 `content_excerpt` 调整为上下文对象或上下文文本。Prompt 中的“正文片段”改为“小说正文上下文”，并根据 `is_full_text` 注明是完整正文还是全书抽样片段。

`source_excerpt_chars` 字段保持不改名，避免数据库迁移和前端兼容成本。它的写入值从“文件开头摘录字符数”调整为 `included_chars`，也就是实际进入 AI 上下文的正文字符数。后续如果需要更准确命名，可以单独迁移为 `source_context_chars`。

旧的 `extract_text_excerpt` 可以保留给元数据生成或其他轻量路径使用，但角色、设定、写作风格分析不再调用它。

## Prompt 原则

所有使用共享上下文的 AI prompt 都应遵守：

1. 明确告诉 AI 当前上下文是完整正文还是全书抽样。
2. 要求只基于提供文本和元数据判断。
3. 抽样上下文下禁止推断未提供的后续剧情、结局和隐藏关系。
4. 对证据字段要求引用提供文本中的短片段。
5. 阅读器助手回答无法确定的问题时，应直接说明“当前上下文无法确认”。

## 测试计划

新增或修改 Python 测试：

1. 短篇上下文构建：正文在预算内时 `is_full_text` 为 true，`context_text` 包含完整正文。
2. 长篇上下文构建：正文超过预算时 `is_full_text` 为 false，包含开头、结尾和中间片段，不再只取开头。
3. 当前章节优先：传入 `focus_chapter_index` 时，该章节进入长篇抽样上下文。
4. 阅读器助手接口：fake AI 收到的消息包含小说上下文、当前章节信息和用户问题。
5. 阅读器助手错误：小说不存在、空问题、未配置 AI 分别返回预期错误。
6. 角色、设定、写作风格分析：fake AI 收到的消息使用新上下文，并且 `source_excerpt_chars` 写入 `included_chars`。

新增或修改 Node 静态 UI 测试：

1. 阅读器模板包含 AI 按钮、助手面板、输入框、发送按钮、清空按钮和状态元素。
2. `reader.js` 包含助手状态、发送函数、渲染函数和新接口路径。
3. `app.js` 绑定助手相关事件。
4. `reader.css` 包含桌面和移动端助手面板布局。

## 风险与缓解

长篇上下文仍然不等于真正全文理解。通过状态提示和 prompt 明确“全书抽样”来降低误导。

上下文字数增加会提高 AI 调用成本和延迟。第一版使用固定字符预算，并复用解析缓存，避免每次重新解析文件。

EPUB 章节解析结果可能包含章节标题或正文提取差异。上下文构建器复用阅读器已验证的 `get_cached_reader_file`，让阅读器和 AI 功能看到同一套文本。

现有 `source_excerpt_chars` 字段名不够准确。第一版保留字段以减少迁移，文案和测试按“实际上下文字符数”理解。

## 验收标准

1. 阅读器页面可以打开 AI 助手，围绕当前小说提问并展示回答。
2. 短篇小说的阅读器助手和分析接口使用完整正文上下文。
3. 长篇小说的阅读器助手和分析接口使用覆盖全书范围的抽样上下文。
4. 当前阅读章节会优先进入阅读器助手上下文。
5. 角色、设定、写作风格分析不再只读取文件开头 12000 字。
6. 未配置 AI 或问题为空时，前后端都有清晰反馈。
7. 相关 Python 和 Node 测试通过。
